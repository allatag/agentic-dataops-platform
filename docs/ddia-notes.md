# DDIA Notes

## Topic: Reliability

### DDIA idea

Reliable systems continue to work correctly even when faults occur. Faults can happen in networks, services, disks, processes, or human operations.

### Application in this project

The ingestion path avoids treating the HTTP request and database write as the only important boundary. Kafka gives the system a durable intermediate log so accepted events can be consumed and persisted independently. The consumer now has explicit behavior for duplicate events, retryable failures, non-retryable poison messages, and DLT routing.

### Design implication

The platform should keep delivery behavior, idempotency, retries, and failure handling explicit before adding higher-level incident context or AI workflows.

### Open question

How should replay be governed so DLT records can be inspected, corrected, and reprocessed without bypassing idempotency?

## Topic: Idempotency

### DDIA idea

At-least-once delivery is the realistic default in distributed systems. Brokers like Kafka may redeliver messages after consumer crashes or rebalances. Idempotency means processing the same message twice produces the same outcome as processing it once.

### Application in this project

The `raw_event` table has a unique constraint on `event_id`. If a duplicate message arrives — due to Kafka redelivery, consumer restart, or manual replay — the consumer catches the `DataIntegrityViolationException` and logs a warning instead of crashing or inserting a duplicate row.

### Design implication

Every event in the system needs a stable, client-generated or deterministic identifier that survives redelivery. The current `eventId` is a UUID generated at ingestion time, which is stable across Kafka retries for the same logical message. Events replayed from a backup or re-published via the API would produce new UUIDs — true deduplication across replay scenarios would require a business-level identifier.

### Open question

Should the idempotency key be the UUID generated by the API (current approach) or a client-provided correlation ID that survives even re-submission from the originating service?

## Topic: Fault Tolerance

### DDIA idea

Fault-tolerant systems expect partial failure and define what happens when a component is temporarily unavailable or an input cannot be processed.

### Application in this project

The raw event consumer distinguishes retryable infrastructure or persistence failures from non-retryable poison messages. Retryable failures use bounded retry and then route valid `RawEvent` records to `raw-events.v1.dlt` if retries are exhausted. Known non-retryable envelope failures, such as unsupported schema versions or missing required fields, go to the DLT without repeated retries.

### Design implication

Failure behavior should be observable and testable. The project now includes failure-mode tests for retry success, retry exhaustion, DLT routing, poison-message classification, and duplicate-event handling.

### Open question

What metadata should be stored with DLT records later so operators can decide whether to replay, discard, or correct a failed event?

## Topic: At-Least-Once Delivery

### DDIA idea

At-least-once delivery favors avoiding message loss, but it means consumers may process the same logical message more than once after retry, restart, rebalance, or offset-commit timing.

### Application in this project

Kafka is the durable handoff between HTTP ingestion and PostgreSQL persistence. The API waits for broker acknowledgement before returning `202 Accepted`, and the consumer handles later persistence. If a message is redelivered, the `raw_event.event_id` unique constraint keeps duplicate storage idempotent.

### Design implication

Consumer logic must treat redelivery as normal. Duplicate detection happens before envelope validation so a replayed duplicate remains a successful skip rather than becoming a poison message.

### Open question

Should future derived projections store their own processed-event markers, or should they rely only on the raw event table for idempotency?

## Topic: Scalability

### DDIA idea

Scalable systems can handle growth in data volume, traffic, or complexity by making bottlenecks visible and designing components that can evolve.

### Application in this project

Kafka separates event intake from persistence work. That gives the project a path to scale consumers, manage backpressure, and add new downstream processors later without changing the initial producer contract.

### Design implication

The raw event topic and database schema should be versioned early enough to support later schema evolution and replay.

### Open question

What partitioning key should be used for raw events so ordering is useful without forcing all events through one partition?

## Topic: Maintainability

### DDIA idea

Maintainable systems are understandable, operable, and adaptable over time. Clear boundaries and explicit decisions reduce accidental complexity.

### Application in this project

The project kept the ingestion and reliability phases focused on one backend, one Kafka-backed ingestion path, and one PostgreSQL persistence boundary instead of introducing agents, RAG, orchestration frameworks, or cloud deployment too early.

### Design implication

Documentation should track architecture decisions as behavior is added, and implementation should prefer simple package boundaries over premature service splits.

### Open question

Which contracts should be documented first: the HTTP request shape, the Kafka event envelope, or the PostgreSQL raw event schema?
